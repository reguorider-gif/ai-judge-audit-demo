不是另一个评分工具。 AI Judge 不替代你现有的 eval 框架。它坐在 eval 的下游，把分数变成人能读懂、能转发、能进入审批流程的审计包。

你缺的不是更多分数，是一份能拿去开会的审计报告。

现在 eval 工具告诉你“faithfulness 低了”。AI Judge 告诉 PM、法务、QA：哪句话有问题、为什么有问题、要不要重跑。

01

给一个失败的回答

RAG 客服回答、agent 自动回复、coding-agent 的 PR 摘要——任何被 eval 判定为不合格的 AI 输出。

02

拆成可验证的主张

把一段话拆成逐条“断言”，每条断言对回源文档核查：有证据、缺证据、还是直接被源文档推翻。

不是 0.51 分。是“C3、C5、C6 三条主张不成立，禁止发布，需重写后重新提交人工审批”。

交付物

审计结果是一份 JSON，不是另一面数据看板。

首页让你理解它做什么。JSON packet 才是产品本体——它可以直接进入 GitHub issue、内部 review queue、合规记录，或者接入你自己 agent 的“要不要继续跑”的判断逻辑。

  • Input: 原始回答、源文档、评测分数、工具调用日志。
  • Audit: 逐条 claim 与证据的对照表、评审分歧记录。
  • Decision: 人工裁决、安全改写版本、下一轮行动指令。

放哪用

接在你已经在用的 eval 和 trace 工具后面。

AI Judge 不替代你的评测框架。它把被标记为失败的案例，包装成产品经理、法务、QA 和工程师都能独立读懂并签字的审计包。

Ragas / TruLens / DeepEval 把 faithfulness 或 groundedness 偏低的结果，转成逐条 claim 的证据审查报告。
Langfuse / Phoenix / Opik / LangSmith 给被标记为异常的 trace，附上一份 reviewer 能直接签字的裁决书。
客服 agent / coding agent / 研究 agent 阻止自动循环一遍遍输出“读起来顺、但事实不可靠”的回答。
